
# RAG Implementation Steps

Very small steps so you can understand and approve each one before it’s implemented. We follow the repo testing rules: regression/feature tests, lowest reliable layer, test behavior not implementation, explain clearly.

**Workflow:** Review a step → approve (e.g. comment “approve step 2”) → implement that step only → you verify → move to next step.

---

# Phase A — Schema and storage

## Step 1: Enable pgvector in Postgres

**What we’re doing**
- Enable the `vector` extension in Postgres so we can store embedding vectors. This is a one-line migration: `CREATE EXTENSION IF NOT EXISTS vector;`.

**Files**
- New migration in `prisma/migrations/` (or a single migration that adds the extension). Prisma doesn’t manage extensions in schema by default, so we may add a raw SQL migration file.

**Test**
- No unit test for the extension itself. We verify by: after migration runs, the DB accepts `vector` type. Optional: a tiny script or test that runs `SELECT 1::vector` (or similar) to confirm extension is loaded. If you prefer to skip a test for this step, we can rely on “migration runs successfully” as the check.

**Approve:** [ ] Step 1 approved

---

## Step 2: Add `QuestionEmbedding` model and table

**What we’re doing**
- Add a new Prisma model `QuestionEmbedding` with: `questionId` (Int, FK to Question), `embedding` (Unsupported("vector(1536)") or use Prisma’s pgvector type if available), `modelVersion` (String, e.g. "text-embedding-3-small"), `updatedAt` (DateTime). One row per question; we upsert on index.
- Create a migration. Note: Prisma 5.x may need `Unsupported("vector(1536)")` for the embedding column unless you use a pgvector Prisma extension.

**Files**
- `prisma/schema.prisma` — add model and relation from Question to QuestionEmbedding (optional 1:1).
- `prisma/migrations/...` — migration generated by `prisma migrate dev`.

**Test**
- Integration-style: after migration, insert a row (e.g. from a test or script) and read it back. Or a minimal test that `prisma.questionEmbedding.create` succeeds with a valid questionId and a placeholder embedding (if Prisma allows). We’re testing “the table exists and we can write/read.”

**Approve:** [ ] Step 2 approved

---

## Step 3: Helper to build embedding input text

**What we’re doing**
- Add a small pure function: given a question’s `question` (text) and `referenceAnswer` (string | null), return the string we will send to the embedding API. Rule: `question + "\n\n" + (referenceAnswer ?? "")` (or agreed format from RAG doc). Trim and handle empty.

**Files**
- New file `lib/embedding-input.ts` (or add to an existing lib file) with `buildEmbeddingInputText(question: string, referenceAnswer: string | null): string`.

**Test**
- Unit test in `lib/embedding-input.test.ts`: (1) question only, (2) question + reference, (3) null reference gives question only, (4) empty string reference. We test behavior: output is always a single string, no undefined, format is consistent.

**Approve:** [ ] Step 3 approved

# Phase B — Calling the embedding API and storing

## Step 4: Call OpenAI Embeddings API and return vector

**What we’re doing**
- Add a function that takes a string, calls OpenAI’s embeddings API (`text-embedding-3-small`), returns an array of numbers (the vector). Same pattern as existing `OPENAI_API_KEY` usage in the app. Handle missing key and API errors.

**Files**
- New file `lib/embeddings.ts` with `getEmbedding(text: string): Promise<number[]>`.

**Test**
- Unit test with mocked OpenAI client: (1) when API returns a vector of length 1536, we get that array; (2) when API throws, we throw or return a clear error. We test behavior: “we get a numeric array of expected length when the API succeeds.”

**Approve:** [ ] Step 4 approved

---

## Step 5: Index one question (build text → embed → upsert)

**What we’re doing**
- Add a function `indexQuestionEmbedding(questionId: number)`: load the question from DB, build input text with `buildEmbeddingInputText`, call `getEmbedding`, upsert into `QuestionEmbedding` (by questionId). Use a fixed `modelVersion` string.

**Files**
- `lib/embeddings.ts` (or `lib/embedding-index.ts`) — add `indexQuestionEmbedding`, using Prisma and `buildEmbeddingInputText` + `getEmbedding`.

**Test**
- Unit test with mocked Prisma and mocked `getEmbedding`: (1) given a question, we call `getEmbedding` once with the correct input string and we upsert one row with that questionId and the mocked vector. We test behavior: “indexing a question results in one upsert with the right questionId and embedding shape.”

**Approve:** [ ] Step 5 approved

---

## Step 6: Backfill script for existing questions

**What we’re doing**
- A script (e.g. `scripts/backfill-question-embeddings.mjs` or `.ts`) that: loads all question ids (or active only), and for each calls `indexQuestionEmbedding`. Rate-limit or small delay if needed to avoid hitting OpenAI limits. No UI change.

**Files**
- New script in `scripts/`. It can import the same `indexQuestionEmbedding` (or a small wrapper that uses Prisma + the lib).

**Test**
- Optional: integration test “run backfill with 0 or 1 question and assert one row in question_embeddings.” Or you approve this step and we verify manually by running the script and checking the table.

**Approve:** [ ] Step 6 approved

# Phase C — Related questions (admin)

## Step 7: API — GET related question slugs for a question id

**What we’re doing**
- New route `GET /api/admin/questions/[id]/related?k=5`. Load the question by id (numeric); require admin. Build embedding input text, call `getEmbedding`, run k-NN against `QuestionEmbedding` (exclude current questionId), return list of question slugs (or ids) for the top k. If the question has no embedding yet, return empty list or 404.

**Files**
- New file `app/api/admin/questions/[id]/related/route.ts`. Use existing admin auth; use Prisma + raw query or Prisma’s vector support for k-NN (e.g. `ORDER BY embedding <-> queryVector LIMIT k`).

**Test**
- API test: mock Prisma so we have two questions with embeddings; request related for question A with k=1; expect response to include question B’s slug. Test behavior: “related endpoint returns slugs of similar questions, not the current one.”

**Approve:** [ ] Step 7 approved

---

## Step 8: Admin question detail — “Related questions” section

**What we’re doing**
- On the admin question detail page (`app/admin/questions/[id]/page.tsx`), add a “Related questions” section. For server-rendered page we can fetch related in the page (call the same k-NN logic or a small internal helper) or add a client component that fetches `GET /api/admin/questions/[id]/related` and displays links to those questions.

**Files**
- `app/admin/questions/[id]/page.tsx` and/or a small client component that calls the related API and renders links.

**Test**
- Page test: mock the data so “related” returns two slugs; render the page; assert the section shows “Related questions” and the two links (or slugs). We test behavior: “when related questions exist, the page shows them as links.”

**Approve:** [ ] Step 8 approved

# Phase D — (Later) Hooks so new/updated questions get embedded

## Step 9: Re-index on question create/update

**What we’re doing**
- When a question is created (POST) or updated (PATCH), after the write, call `indexQuestionEmbedding(questionId)` so the new or updated text is reflected in the embedding table. Do this in the API route or via a small shared function. Avoid blocking the HTTP response too long; we can fire-and-forget or queue.

**Files**
- `app/api/admin/questions/route.ts` (create) and `app/api/admin/questions/[id]/route.ts` (update). After `prisma.question.create` or `prisma.question.update`, call the index function.

**Test**
- API test: create or update a question, then assert that `QuestionEmbedding` has a row for that questionId (and optionally that the embedding is non-null). We test behavior: “creating/updating a question results in an embedding row.”

**Approve:** [ ] Step 9 approved

# After you approve

Once you approve a step, implementation will:

1. Implement only that step.
2. Add or update the test as described (following testing rules: behavior, lowest reliable layer, clear explanation).
3. Leave the next step unchecked until you approve it.

You can approve steps one at a time or in a small batch (e.g. “approve steps 1–3”); we’ll still implement and test one step at a time so you can verify each.
